# 甄嬛传题目与匹配算法说明

这份文档说明当前项目里「题目设计 -> 分数计算 -> 角色匹配」的完整逻辑，便于后续校准题库、解释结果与排查问题。

## 1. 总体流程

1. 用户答题（每题 4 个李克特选项）。
2. 每次作答写入一个 `scoreChanges`（五维增减向量）。
3. 全部题目作答后，把所有 `scoreChanges` 累加到五维基准分上。
4. 得到最终五维分（0~100）。
5. 用角色模板 `stats` 与用户五维分做匹配排序，取第一名作为结果角色。

---

## 2. 维度定义与基准分

### 2.1 甄嬛传五维（顺序固定）

- `emotion`：情感
- `strategy`：心机
- `control`：控制
- `resilience`：韧性
- `courage`：胆识

### 2.2 基准分

所有维度初始值都是 `50`。

- 实现位置：`src/domain/scoring/score-calculator.js`
- 逻辑：`buildBaseScores()` 返回 `[50, 50, 50, 50, 50]`

---

## 3. 题目结构与选项逻辑

题库分为两类：`anchor`（锚题）和 `cross`（交叉题）。

### 3.1 选项强度（统一）

每题都是 4 档同意度：

- A: 非常同意 -> `intensity = 2`
- B: 同意 -> `intensity = 1`
- C: 不同意 -> `intensity = -1`
- D: 非常不同意 -> `intensity = -2`

> 注意：A/B/C/D 不是固定“加分/减分”，是否加减还取决于题目方向与 `crossCut`。

### 3.2 锚题（anchor）怎么计分

锚题只打一个主维度（`primary`）。

公式：

`scoreChanges[primary] = intensity * 2 * direction`

- `direction = 1`：同意越高，主维越高
- `direction = -1`：同意越高，主维越低（反向题）

所以锚题对主维单题影响幅度为：`±4` / `±2`。

### 3.3 交叉题（cross）怎么计分

交叉题同时影响两个维度：`primary` 和 `secondary`。

公式：

- `primary`：`intensity * 2 * crossCut[primary]`
- `secondary`：`intensity * 1 * crossCut[secondary]`

其中 `crossCut` 的值通常是 `1` 或 `-1`，用于表达题意方向。

因此交叉题常见影响是：

- 主维：`±4` / `±2`
- 次维：`±2` / `±1`

---

## 4. 答题后如何累计成最终五维分

### 4.1 单题作答时

每答一题，会把该题选项的 `scoreChanges` 存入 `selectedAnswers[currentQuestionIndex]`。

- 实现位置：`src/app/usecases/answer-question.js`

### 4.2 交卷时

系统将所有已选答案的 `scoreChanges` 按维度累加到基准分上：

`finalScore[dim] = clamp( baseScore[dim] + Σ(scoreChanges[dim]) )`

并进行边界裁剪：

- 小于 0 -> 0
- 大于 100 -> 100

## 5. 角色匹配算法（核心）

### 5.1 候选角色

从 `characterMap` 读取所有角色模板 `stats`。  
仅保留维度长度与用户分数长度一致的角色。

### 5.2 计算三项指标

对每个角色计算：

1. **shapeSimilarity（形状相似度）**  
   - 把每一维都先减去 50 再做余弦相似度（看“形状方向”是否一致）。
2. **distance（距离）**  
   - 加权曼哈顿距离：`Σ |candidate[i]-user[i]| * weight[i]`
   - 权重来自当前剧本 `manifest.dimensions[].weight`。
3. **maxDifference（最大单维差）**  
   - 五维中最大那一维的绝对差。

### 5.3 排序规则（非常关键）

按以下优先级排序：

1. `distance` 越小越优先（升序）
2. `shapeSimilarity` 越大越优先（降序）
3. `maxDifference` 越小越优先（升序）
4. 仍相同则按角色 key 字典序

> 当前实现先看“绝对分差近不近”，再看“人格轮廓方向像不像”。  
> 这么做的目的是降低边界型结果占比，减少“top1 与 top2 太接近”的碰撞感。

### 5.4 产出结果

排序第一名角色作为 `finalProfile`，用于结果页头像、名字、台词、文案与主题色。

---

## 6. 为什么会出现“不同人但五维图看起来接近”

这通常不是代码错误，而是以下机制共同造成的表现：

1. 初始都从 50 开始，题目增减大多是中等幅度；
2. 很多用户答题习惯会落在中间区间；
3. 匹配优先看“形状相似”，可能出现多个角色轮廓都接近；
4. 结果页默认只展示用户图，不展示“与角色模板的差值”。

## 7. 当前实现中的注意点

1. `executeSubmitQuiz` 默认 `fallbackKey = "liwei"`（更偏雍正剧本）。  
   在甄嬛传 `characterMap` 中通常没有这个 key，因此兜底可能为 `null`（但正常情况下都有匹配结果，不影响主流程）。

2. 头像显示依赖 `finalProfile.avatarUrl`。  
   如果资源文件后缀与路径不一致，会触发图片加载失败（目前结果页已有后缀兜底尝试逻辑）。

## 9. 给运营/产品的解读话术（可直接用）

- 这套系统不是“单题对应角色”，而是 40 题共同形成五维人格轮廓后，再与角色模板比相似度。
- 结果角色是“最接近你整体轮廓的人”，不代表你与该角色完全相同。
- 当第一名和第二名接近时，说明你处于两种人格风格的边界区间，属于正常现象。
